# How It Works

A technical deep-dive into the ESP32 Sensor Dashboard architecture — for the curious, the makers who want to extend it, and anyone who's ever wondered what's actually happening between the sensor and the browser.

---

## Firmware side (main.cpp)

### Startup sequence

```
setup()
  │
  ├─ Serial.begin(115200)
  ├─ Wire.begin()                ← I²C on SDA=21, SCL=22
  ├─ analogReadResolution(12)    ← 12-bit ADC = 0 to 4095
  ├─ analogSetAttenuation(11dB)  ← full 0–3.3V input range
  ├─ LittleFS.begin()            ← mount flash filesystem
  ├─ loadConfig()                ← restore sensor config from NVS
  ├─ setupWiFi()                 ← try STA, always start AP
  ├─ setupServer()               ← register WebSocket + HTTP routes
  └─ printPinMap()               ← log sensor → GPIO table to Serial
```

### The sensor config system

Sensors are stored as an array of `SensorConfig` structs in RAM (up to 12 simultaneous sensors):

```cpp
struct SensorConfig {
  String id, label, type;   // e.g. "s_abc123", "Soil Moisture", "soil"
  SensorKind kind;           // enum: KIND_ANALOG, KIND_DHT11, KIND_ULTRASONIC…
  int pin, pin2;             // GPIO numbers (pin2 only used for ultrasonic)
  float warnLow, warnHigh;   // amber threshold range
  float dangerLow, dangerHigh; // red threshold range
  bool enabled;
};
```

Config is serialised to JSON and stored in NVS using `Preferences`. On every boot, `loadConfig()` restores it and calls `applySensorPins()` which:
- Sets `pinMode` on every configured GPIO
- Allocates DHT objects for DHT11/22 sensors (they need an object to hold state)

### Reading sensors

Every 1000ms, `loop()` calls `readSensor(i)` for each configured sensor. The return type is a `SensorReading` struct:

```cpp
struct SensorReading {
  String id, label, type;
  String unit, unit2;        // e.g. "°C", "%"
  float value, value2;       // value2 used for DHT temp+humidity
  String status;             // "ok", "warning", "danger", "offline"
  String analysis;           // human text: "normal", "dry", "CO detected"
  bool healthy;
};
```

The `status` field is computed by comparing the value against the sensor's configured thresholds.

The `analysis` field is generated by `analysisFor()` — a simple lookup that returns a human-readable description based on sensor type and value:

```cpp
if (cfg.type == "soil") {
  if (value < 35) return "dry";
  if (value > 75) return "saturated";
  return "good";
}
```

### WebSocket communication

All live communication uses a single persistent WebSocket connection at `/ws`. The ESP32 uses `ESPAsyncWebServer` + `AsyncWebSocket` — both run on the async task scheduler, meaning they never block the main loop.

**ESP32 → Browser (broadcast, every second):**

```json
{
  "type": "sensor_update",
  "timeMs": 45231,
  "sensors": [
    {
      "id": "s_abc123",
      "label": "Soil Moisture",
      "type": "soil",
      "value": "62.40",
      "value2": null,
      "unit": "%",
      "unit2": "",
      "status": "ok",
      "analysis": "good",
      "healthy": true
    }
  ]
}
```

**ESP32 → Browser (broadcast, every 3 seconds):**

```json
{
  "type": "device",
  "uptimeMs": 45000,
  "freeHeap": 223412,
  "ip": "192.168.1.9",
  "clients": 2,
  "wifiConnected": true
}
```

**Browser → ESP32 (on connect):**

```json
{ "type": "get_config" }
```

**ESP32 → Browser (response to get_config):**

```json
{
  "type": "config",
  "config": {
    "sensors": [
      {
        "id": "s_abc123",
        "label": "Soil Moisture",
        "type": "soil",
        "pin": 34,
        "pin2": -1,
        "enabled": true,
        "warnHigh": 80,
        "dangerHigh": 95
      }
    ]
  }
}
```

**Browser → ESP32 (saving new config):**

```json
{
  "type": "save_config",
  "config": { "sensors": [...] }
}
```

On receiving `save_config`, the ESP32:
1. Parses and validates the JSON
2. Updates `sensors[]` in RAM
3. Calls `applySensorPins()` to reconfigure GPIOs
4. Saves to NVS flash
5. Broadcasts the new config back to all clients
6. Broadcasts an immediate sensor reading

**Browser → ESP32 (relay toggle):**

```json
{ "type": "relay", "id": "s_relay1", "state": true }
```

**Browser → ESP32 (I²C scan):**

```json
{ "type": "i2c_scan" }
```

**Browser → ESP32 (pin probe):**

```json
{ "type": "pin_probe" }
```

### Pin Probe — how it works

When the browser requests a pin probe, the firmware:

1. Iterates every safe **ADC1 pin** (GPIO 32, 33, 34, 35, 36, 39)
2. Takes 3 analog readings with 5ms gaps, computes average and **range** (max − min)
3. Marks a pin "active" if `average > 50` OR `range > 30` (variation = live signal)
4. Iterates remaining safe **digital pins** (GPIO 2, 4, 5, 12–27 minus ADC1 pins)
5. Calls `digitalRead()` on each and marks active if HIGH

The range threshold is the key insight: an unconnected floating pin also reads non-zero, but it reads the *same* non-zero value each time. A live sensor will have a non-zero **range** between samples because the signal is actually changing.

### WiFi strategy

```
setupWiFi()
  │
  ├─ WiFi.mode(WIFI_AP_STA)           ← both modes simultaneously
  │
  ├─ if (WIFI_SSID is set)
  │   ├─ WiFi.begin(SSID, PASS)
  │   ├─ wait up to 12 seconds
  │   └─ print result to Serial
  │
  └─ WiFi.softAP(AP_SSID, AP_PASS)   ← always runs, regardless of STA result
```

The fallback AP (`ESP32-Dashboard`) always runs. If the ESP32 also connects to a router, it's accessible on both IPs simultaneously.

## Web files side (data/)

The HTML, CSS, and JavaScript live on the ESP32's LittleFS filesystem, uploaded separately from the firmware with `pio run --target uploadfs`.

When a browser hits `http://[ip]/`, the ESP32 reads `index.html` from LittleFS and sends it. The browser then requests `style.css` and `app.js` — also served from LittleFS.

### app.js — state machine

The frontend is a single-file vanilla JS application with no frameworks or build tools:

```
state = {
  ws,           ← WebSocket connection
  config,       ← sensor config received from ESP32
  latest,       ← Map: sensor ID → most recent SensorReading
  history,      ← Map: sensor ID → last 72 values (for sparklines)
  dark          ← current theme
}
```

**WebSocket lifecycle:**
```
connect()
  → ws.onopen: send get_config
  → ws.onmessage: dispatch to handleMessage()
  → ws.onclose: schedule reconnect in 1.5s
  → ws.onerror: close (triggers onclose)
```

**On receiving config:**
```
handleMessage({ type: "config" })
  → state.config = message.config
  → renderConfig()     ← rebuilds the Setup panel form rows
  → renderDashboard()  ← builds the sensor card grid
```

**On receiving sensor_update (every second):**
```
handleMessage({ type: "sensor_update" })
  → for each sensor reading:
      state.latest.set(id, reading)
      state.history.get(id).push(value)  ← rolling 72-point buffer
      update card DOM (status, value, unit, analysis)
      drawSparkline(canvas, history[id], status)
```

### Sparkline rendering

Each sensor card contains a `<canvas>` element. Every second, `drawSparkline()` draws the history as a line chart:

1. Compute min/max of the history values
2. Normalise each value to canvas height with 12% vertical padding
3. Choose line colour from the current status: `--ok`, `--warn`, `--danger`, `--offline`
4. Draw grid lines from the CSS `--border` variable (so it respects dark mode)
5. Stroke the line with `lineJoin: "round"` for smooth curves

Canvas dimensions are multiplied by `devicePixelRatio` to stay sharp on retina / high-DPI screens.

### Sensor catalog

The sensor catalog in `app.js` is a plain JavaScript array of objects:

```js
const CATALOG = [
  {
    name: "DHT22 / AM2302",
    category: "Temp / Humidity",
    label: "DHT22 Temp+Humidity",
    type: "dht22",
    desc: "More accurate than DHT11, same wiring",
    wiring: "VCC→3.3V · GND→GND · DATA→any digital GPIO + 10kΩ pull-up",
    pin: 4,
    pin2: -1,
    warnHigh: 35,
    dangerHigh: 42,
  },
  // ... 24 more entries
];
```

When a user clicks a sensor card in the library:
1. `addFromCatalog(entry)` creates a new sensor config object from the catalog entry
2. Pushes it to `state.config.sensors`
3. Calls `renderConfig()` to add the pre-filled form row
4. Shows the wiring note in the hint area
5. The user adjusts the GPIO pin and clicks Save

Nothing is sent to the ESP32 until the user explicitly clicks **Save config**.

## Data flow summary

```
Physical sensor
      │
      │ voltage / current
      ▼
ESP32 GPIO
      │
      │ analogRead() / digitalRead() / DHT.read() / pulseIn()
      ▼
readSensor()  →  SensorReading { value, status, analysis }
      │
      │ JSON serialisation
      ▼
WebSocket broadcast  →  every connected browser
      │
      │ JSON.parse()
      ▼
updateReadings()  →  update card DOM + push to history[]
      │
      │ canvas 2D API
      ▼
drawSparkline()  →  60-second rolling line chart
```

## Memory usage

On a typical ESP32 with 2–4 sensors configured:

| Resource | Usage |
|---|---|
| Flash (firmware) | ~920KB of 1.2MB available |
| Flash (LittleFS) | ~15KB (HTML + CSS + JS) |
| RAM | ~45KB of 320KB used |
| CSV log buffer | Up to 32KB in RAM, then stops growing |

The 32KB CSV log cap in RAM is intentional — the ESP32 doesn't have unlimited heap. For longer logging, export and clear the log periodically.

## Extending the project

### Add a new sensor type to the firmware

1. Add a new `KIND_*` value to the `SensorKind` enum
2. Handle it in `kindFromType()`, `applySensorPins()`, and `readSensor()`
3. Add a case in `analysisFor()` for human-readable output
4. Add a case in `unitForType()` for the unit string

### Add a sensor to the browser catalog

In `app.js`, add an object to the `CATALOG` array:

```js
{
  name: "Your Sensor Name",
  category: "Your Category",
  label: "Display Label",
  type: "analog",          // must match a firmware type
  desc: "One line about what it measures",
  wiring: "VCC→3.3V · GND→GND · OUT→ADC pin 32-39",
  pin: 34,                 // suggested GPIO
  pin2: -1,
  warnHigh: 2000,
  dangerHigh: 3500,
}
```

That's it — no firmware change required for sensors that map to existing types.

### Add an HTTP API endpoint

In `setupServer()` in `main.cpp`:

```cpp
server.on("/api/your-endpoint", HTTP_GET, [](AsyncWebServerRequest *r) {
  r->send(200, "application/json", yourJsonFunction());
});
```

### Add a WebSocket message type

In `handleWsMessage()`:

```cpp
} else if (type == "your_command") {
  // do something
  client->text(yourResponseJson());
}
```

## Libraries used

| Library | Purpose |
|---|---|
| `ESPAsyncWebServer` | Non-blocking HTTP + WebSocket server |
| `AsyncTCP` | Async TCP layer (dependency of ESPAsyncWebServer) |
| `ArduinoJson` v7 | JSON serialisation / deserialisation |
| `DHT sensor library` (Adafruit) | DHT11 / DHT22 reading |
| `Adafruit Unified Sensor` | Dependency of the DHT library |

| `Preferences` (built-in) | Non-volatile key-value storage for config |
| `WiFi` (built-in) | WiFi STA + AP mode |
